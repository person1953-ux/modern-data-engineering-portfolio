Databricks can run in parallel, but it is not fully automatic for your case — you must set limits.
For Jobs API (run-now), parallelism is controlled by the job setting Maximum concurrent runs.
If it is 1, runs are serialized (one-by-one).
If you set it to 20, up to 20 run in parallel and others queue.
Actual achieved parallelism also depends on compute capacity and workspace quotas.
So: Databricks supports parallel, but you need to configure it explicitly in Job settings.


### As-Is (Tibco RV + BW)
- ~20 BW runner engines subscribe to same RV messages.
- Multiple engines receive same event, but schedule logic allows only eligible execution. Schedule logic sometimes broken due to network issues (message loss) resulting duplication and table locked due to more than one BW engine inserting data to table at the same time. 
- Capacity is mostly pre-provisioned (paid even during idle periods).
- Peak throughput observed at ~20 concurrent jobs.
- Practical data volume ceiling observed around 1 GB per job.
- Configuration is fragmented and largely manual for new ETL pipelines (source/target DB connection, SQL setup, and limited rerun handling on failure).

### To-Be (Airflow + Databricks)

- Airflow controls orchestration and schedule logic centrally.
- Databricks executes ETL workloads with configurable parallelism.
- New ETL onboarding is metadata-driven (100% automation) :  Add a row in ETL meta table, and the DAG automatically picks up the configuration and triggers execution without additional pipeline code changes.
- Elastic compute: pay mostly for execution windows, not idle listeners.
- Better task-level observability, retry, and lineage than listener-style fanout.
- Target throughput of 50+ concurrent jobs (based on configured orchestration and compute scaling).
- Can processed 100+ GB workloads on Databricks/Spark using Delta Lake, optimized partitioning, caching, and autoscaling — aligned with modern AI/ML requirements including: large-scale feature engineering,training data preparation for LLMs, distributed model training support, real-time inference data pipelines

1. Oracle RAC
Cost: Very high (Oracle licenses, RAC option, hardware).
IT Infrastructure: Complex (shared storage, cluster setup, Oracle-specific drivers).
Docker/Airflow: Possible, but setup is complex and not cloud-native.
Summary: Not recommended for low-cost, simple, or containerized environments.
2. MySQL Cluster
Cost: Lower than Oracle RAC (open-source, but requires multiple nodes for HA).
IT Infrastructure: Moderate (multiple MySQL nodes, network config, NDB engine).
Docker/Airflow: Works with Docker, but cluster setup is more complex than single-node MySQL. Airflow integration is easy (standard MySQL drivers).
Summary: Cheaper and simpler than RAC, but still requires cluster management.
3. Distributed ETL Engines (e.g., PySpark, Dask)
Cost: Low to moderate (open-source, pay for compute only if using cloud).
IT Infrastructure: Minimal for small/medium jobs (can run single-node Spark/Dask in Docker). For large scale, need cluster setup.
Docker/Airflow: Excellent compatibility. PySpark and Dask run well in Docker and are easily orchestrated by Airflow (via BashOperator, SparkSubmitOperator, etc.).
Summary: Easiest to containerize, lowest infrastructure for ETL, and flexible for scaling.
Recommendation
For lowest cost, minimal IT setup, and best Docker/Airflow compatibility:
Distributed ETL engines (PySpark, Dask) are the best choice.

You only need Python, the ETL library, and standard DB drivers (JDBC for Spark, or Python DB-API for Dask).
No need for complex DB clustering.
Easy to run in Docker and orchestrate with Airflow.
